# Spec: Book Editor

Feature: an editing stage between "Claude generated a layout" and "export the
PDF." The owner reviews the AI draft, then edits it directly or by asking the
AI, page by page or book-wide. Builds entirely on the existing layout_spec
JSONB, BookPages renderer, and export pipeline; the PDF path does not change.

## Product goals

1. The AI draft is a starting point, not the final say. Every AI decision
   (photo selection, page template, caption, narrative) is human-overridable.
2. Curation is a feature: the book should be the best 40 photos, not all 400.
   The editor makes excluding photos easy and shows what was left out.
3. Editing works two ways, always both available:
   - Direct manipulation: tap a page, swap a photo, retype a caption.
   - AI instruction: "make day 3 shorter", "write a funnier caption",
     "add a page for the lighthouse photos."
4. Nothing is destructive. Every AI pass and manual save is a revision the
   owner can roll back.

## Decisions (locked)

- D1: The editor edits a **draft**, a new row type, not the export. Exports
  become immutable snapshots of a draft at render time. Rationale: today
  layout_spec lives on book_exports; regenerating stomps edits. Drafts
  separate "the working book" from "a PDF I made Tuesday."
- D2: **One draft per trip** in v1. Revisions give history; parallel drafts
  add UI complexity with no family-scale payoff.
- D3: The layout spec schema is extended, not replaced. Existing renderer
  keeps working; new fields are optional.
- D4: Editor is owner-only for writes in v1, members can view. Matches the
  existing export permission model. Multi-editor merge is out of scope.
- D5: AI edits are expressed as **JSON Patch-style operations against the
  spec**, generated by Claude via tool use, validated server-side, then
  applied. Claude never returns a whole rewritten book for a small ask.
  Rationale: cheap, fast, diffable, and a hallucinated photoId in one op
  can be rejected without losing the rest.
- D6: Photo truth stays in Postgres; the spec references photoIds only.
  The "photo tray" (unused photos) is computed: trip photos minus photoIds
  present in the spec. No duplicate bookkeeping.
- D7: Responsive editor, same PWA, two layouts from one codebase. Capture
  and journaling are phone-first; book editing and formatting are
  laptop-first. Below 1024px: single column, photo tray and assistant as
  bottom sheets. At 1024px and up: two-pane layout with the spread view
  center and a sticky right sidebar (tabs: Photos / Assistant / History),
  larger page previews, and the same controls. Reordering uses explicit
  move controls on both (works everywhere, no drag-and-drop dependency).
- D8: Text formatting is intentionally minimal: per-page template choice,
  caption text, body text, and a per-chapter title. No fonts, colors, or
  free positioning in v1. The book's typography is the design system's job;
  consistency is what makes it look professionally printed.
- D9: Revisions are full spec snapshots (JSONB), capped at the last 20.
  Specs are tens of KB; snapshotting is simpler and safer than op logs.

## Schema changes

```sql
CREATE TABLE book_drafts (
  id BIGSERIAL PRIMARY KEY,
  trip_id BIGINT NOT NULL UNIQUE REFERENCES trips(id) ON DELETE CASCADE,
  spec JSONB NOT NULL,
  mode TEXT NOT NULL,                      -- auto | curated (last generation mode)
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE book_draft_revisions (
  id BIGSERIAL PRIMARY KEY,
  draft_id BIGINT NOT NULL REFERENCES book_drafts(id) ON DELETE CASCADE,
  spec JSONB NOT NULL,
  source TEXT NOT NULL,                    -- 'generate' | 'manual' | 'ai-edit' | 'restore'
  note TEXT,                               -- e.g. the AI instruction that produced it
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE book_exports ADD COLUMN draft_id BIGINT REFERENCES book_drafts(id);
-- book_exports.layout_spec remains: the frozen snapshot actually rendered.
```

Generation flow change: "Generate book" now upserts the draft (writing a
'generate' revision first if a spec existed). "Export PDF" copies
draft.spec into a new book_exports row and renders that copy.

## Spec schema v2 (extends v1, renderer-compatible)

```
{
  "version": 2,
  "title": str, "subtitle": str,
  "chapters": [{
    "id": "ch_<uuid>",                 // stable ids so ops can address parts
    "title": str,
    "narrative": str,
    "pages": [{
      "id": "pg_<uuid>",
      "template": "full-bleed | two-up | three-grid | photo-text | text-only",
      "photoIds": [int],
      "caption": str,
      "text": str,
      "pinned": bool                   // true = AI passes must not modify this page
    }]
  }],
  "excludedPhotoIds": [int]            // owner said "never include"; AI respects it
}
```

Migration: the server upgrades v1 specs lazily on first draft load by
assigning ids and version 2.

## Editor UI

Route: /trip/[id]/book/edit. Three working surfaces:

1. **Spread view** (default): vertically scrolled pages exactly as the
   preview renders them (reuses BookPages), each page wrapped in an edit
   frame with: template switcher, caption/text fields (inline, tap to edit),
   per-photo controls (replace from tray, remove), page actions (move up,
   move down, duplicate, delete, pin, "AI: redo this page" with an optional
   instruction box), and an "insert page here" affordance between pages.
2. **Photo tray**: bottom sheet listing every ready trip photo, grouped by
   day, badged as In book / Unused / Excluded. Tap a photo to place it on
   the selected page (template auto-upgrades: full-bleed -> two-up ->
   three-grid; a fourth photo requires a new page). Long-press to Exclude.
   The tray is where curation happens: the count chip reads like
   "58 photos, 34 in book, 6 excluded."
3. **Book chat**: a persistent instruction box (and message history for the
   session) for book-level asks: "tighten day 2 into one chapter",
   "the tone is too flowery, make it plainer", "find the 10 best beach
   photos and give them their own chapter." Each AI response applies as one
   revision with the instruction saved as the revision note, and the spread
   view scrolls to the first changed page with changed pages briefly
   highlighted.

Toolbar: Undo (restore previous revision), History (list revisions, tap to
preview, Restore), Preview (clean render, no edit chrome), Export PDF.

Autosave: manual edits debounce-save after 1.5s idle; a 'manual' revision is
cut at most once per 5 minutes of continuous editing or on leaving the page,
so undo granularity stays useful.

## API surface

```
GET    /api/trips/:id/draft                  -> { draft, revisions: [meta] }
PUT    /api/trips/:id/draft                  -> save spec (owner; validates schema + photoIds)
POST   /api/trips/:id/draft/generate         -> full (re)generation, mode in body (existing pipeline, now targets draft)
POST   /api/trips/:id/draft/ai-edit          -> { instruction, scope? { chapterId?, pageId? } }
POST   /api/trips/:id/draft/revisions/:rid/restore
POST   /api/trips/:id/exports                -> unchanged externally; now snapshots the draft
```

ai-edit is synchronous up to 60s (page/chapter scope) and falls back to the
existing async status-poll pattern for whole-book instructions.

## AI integration design

One endpoint, three internal modes chosen by scope:

- **Page scope**: prompt contains the page, its chapter context, the photo
  manifest of in-book and tray photos for that day, and the instruction.
- **Chapter scope**: the chapter plus day's tray photos.
- **Book scope**: full spec plus full manifest (same corpus builder as
  generation).

Claude is called with a single tool, `apply_edits`, and must respond with a
tool call:

```
apply_edits({
  ops: [
    { op: "replace_page", pageId, page: {...} },
    { op: "insert_page", chapterId, afterPageId | null, page: {...} },
    { op: "remove_page", pageId },
    { op: "move_page", pageId, chapterId, afterPageId | null },
    { op: "set_chapter", chapterId, title?, narrative? },
    { op: "insert_chapter", afterChapterId | null, chapter: {...} },
    { op: "remove_chapter", chapterId },
    { op: "set_meta", title?, subtitle? },
    { op: "exclude_photos", photoIds: [...] },
    { op: "include_photos", photoIds: [...] }   // pulls from excluded back to tray
  ],
  summary: "one sentence shown to the user"
})
```

Server-side validation before applying, all-or-nothing per op but not per
batch (invalid ops are dropped and reported, valid ops apply):

- Every referenced photoId exists, is 'ready', belongs to the trip, is not
  in excludedPhotoIds, and is not already used on another page (unless the
  same op removes it there).
- pinned pages reject modification ops.
- Template/photo-count coherence (two-up has exactly 2, etc.).
- Curated-mode drafts keep the curated system prompt: AI ops may not
  introduce text that is not drawn from entries.

Guardrails carried over from generation: the corpus is the only source of
truth; the system prompt for page/chapter scope explicitly forbids inventing
events. The `summary` string plus the op list gives the revision note.

### Curation intelligence

"Not every picture needs to be in the book" gets explicit support:

- Generation prompt (both modes) is updated from "aim to include every
  photo" to: select for the story; prefer one strong photo over three near
  duplicates (same author within 2 minutes at the same place); target a
  book of 20 to 40 pages for a week-long trip; never use excludedPhotoIds.
- A dedicated quick action in the tray: "AI curate" runs a book-scope edit
  with the instruction template "review unused and in-book photos; swap in
  stronger ones, drop weak or duplicate ones, keep the page count similar."
- The AI sees only metadata (time, place, author, orientation, attached
  entry text), not pixels, in v1. Vision-based quality scoring (sharpness,
  faces, duplicates by content) is a listed v2 upgrade: run Claude vision
  over preview images at generation time, store a per-photo score and
  duplicate-group id, and feed both into the manifest. The manifest field
  (`quality`, `dupGroup`) is reserved in the schema now so v2 is additive.

## What v1 explicitly does not do

- No free-form page layout (drag photos around a canvas).
- No font/color/theme options.
- No collaborative editing or member suggestions.
- No image editing (crop/rotate happens implicitly via object-fit cover;
  a manual focal-point picker is a v2 candidate).
- No print-service integration; the deliverable is the PDF.
